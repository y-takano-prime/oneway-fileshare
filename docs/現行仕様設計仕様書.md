## 5. コントローラー設計／処理仕様

業務ルールの背景は`現行仕様要件定義書.md` 8章を参照。本章は処理手順・バリデーション文字列・内部実装手段のみを記載する。

### 5.1 Auth\LoginController

- `showLoginForm()`：ログイン画面表示
- `login(Request)`：バリデーション `email`(required, email, max:255) / `password`(required, string, min:4)。`Auth::attempt()`で照合。失敗時は`withErrors(['email' => 'メールアドレスまたはパスワードが正しくありません'])`（入力欄はemailのみ復元）
- `is_active=false`のユーザーは認証成功後に即`Auth::logout()`し、`withErrors(['email' => 'このアカウントは無効化されています'])`を設定
- ビュー（`auth/login.blade.php`）は`{{ $errors->first('email') }}`でこのメッセージを表示（2026-06-22修正：以前は固定文字列のみで上記メッセージが画面に出ない不整合があったが解消済み）
- ログイン成功時：`$request->session()->regenerate()`（セッション固定化対策）→ `intended()`または`/dashboard`へリダイレクト
- `logout()`：セッション無効化＋CSRFトークン再生成

### 5.2 DashboardController

- `index()`：`DownloadUrl`を管理者は全件、担当者は`user_id`で絞り込み取得
- 計算するメトリクス：DL済み件数（`expires_at`未来かつ`download_count>0`）／未DL件数（`expires_at`未来かつ`download_count=0`）／期限切れ件数（`expires_at`過去）／無効化済み件数（`onlyTrashed()`）／ストレージ使用量・上限・割合
- `recentUrls`：最新10件
- **注意：** `$totalUrls`・`$totalDownloads`・`$validCount`・`$totalSize`も計算しているが、現在のビューはいずれも使用していない（死んでいる変数）。再構築時はビューで使う値のみ計算すればよい

### 5.3 FileController

- `index()`：担当者は自分のファイルのみ、管理者は全担当者分。検索`q`（ファイル名）。ソート可能列：ファイル名・URL数・アップロード者・サイズ・アップロード日時・`earliest_expires_at`（紐づくURLのうち最も早い`expires_at`。画面表示は猶予日数を加算した「削除予定日」だが、ソートキー自体には猶予日数は反映されない）
- `destroy(SharedFile $file)`：所有者本人または管理者のみ可（インラインチェック、`authorizeOwner()`とは別実装）。即時・物理削除（猶予期間なし）。紐づく全`DownloadUrl`をソフトデリート。`deleted_files_log`に記録
- 複数ファイル一括アップロード用の`store()`は2026年6月22日に削除済み（アップロード入口は5.4のステップ2に一本化）

### 5.4 DownloadUrlController

3ステップのURL発行フローを担当。

- `storeStep1(Request)`：バリデーション
  ```
  category         : required, in:business,recruitment,other
  company_name     : nullable, string, max:255
  recipient_title  : nullable, string, max:255
  recipient_name   : required, string, max:255
  recipient_email  : required, email, max:255
  expires_at       : required, date, after:now
  download_limit   : nullable, integer, min:1, max:9999
  memo             : nullable, string, max:2000
  notify_on_download: boolean（$request->boolean()で変換）
  ```
  入力値はDBに保存せず、セッションキー`create_recipient`に一時保持して`urls.create_step2`へリダイレクト。管理者がこのアクションを呼ぶと403「URL発行は担当者のみ操作できます」
- `create(Request)`：クエリパラメータ`shared_file_id`を受け取り、自分のファイルか検証の上で隠しフィールド→セッション`create_preselected_file_id`に引き継ぐ（2026-06-22修正：以前はこの値が使われておらず事前選択が機能していなかった）
- `createStep2()`：セッションに`create_recipient`が無ければ`urls.create`へリダイレクト。自分（`Auth::id()`）がアップロード済みの`SharedFile`一覧（`latest()`順）を表示。`create_preselected_file_id`があればラジオボタンを事前チェック
  - 既存ファイル選択時：`shared_file_id`バリデーション `required`, `Rule::exists('shared_files','id')->where('user_id', Auth::id())`。エラー文言「ファイルをアップロードするか、既存のファイルを選択してください。」
  - 新規アップロード時：`upload_file`バリデーション `required, file, max:102400, mimes:jpg,jpeg,png,gif,pdf,doc,docx,xls,xlsx,ppt,pptx,zip,csv,txt`。保存先`storage/app/uploads/{user_id}/{UUID}.{拡張子}`（`local`ディスク）。アップロード成功時、即座に`category=null`の`SharedFile`を作成
- `store(Request)`：`DownloadUrl`を作成。`token`は`Str::random(64)`、`download_count`は0で初期化、ステップ1のセッション値とマージして保存。保存後セッションの`create_recipient`を破棄し、`urls.complete`へリダイレクト
- `complete(DownloadUrl $url)`：**`authorizeOwner()`必須**（2026-06-22修正：欠落していたIDOR脆弱性を修正済み。他担当者のURLをID指定で開くと、OTP認証前の生ダウンロードトークン入りメール文面が閲覧できてしまう状態だった）。`buildMailBody()`で送付メール文章を生成して表示（文章構造は6章参照）
- `index()`：検索は「相手先（`q`：相手先名・相手先メール・ファイル名のいずれかにLIKE一致）」と「担当者名（`staff_q`、管理者専用）」の2系統に分離（同一人物が相手先・担当者の両方に存在する場合の誤マッチ防止のため統合しない）。担当者名検索は`REPLACE(REPLACE(name, ' ', ''), '　', '') LIKE ?`で全角/半角スペース無視。状態フィルター（複数選択、OR）：`wait`(未DL=期限内かつdownload_count=0) / `done`(DL済み=期限内かつdownload_count>0) / `expired` / `invalidated`。属性フィルター（複数選択、OR）：`business`/`recruitment`/`other`。ソート可能列：`created_at`/`expires_at`/`download_count`/`recipient_name`/`category`。ページサイズ20件。`withTrashed()`で無効化済みも一覧表示
- `show(DownloadUrl $url)` / `edit` / `update` / `destroy` / `sendMail`：いずれも`authorizeOwner()`必須
- `sendMail(DownloadUrl $url)`：担当者自身のメールアドレスを送信元として`FileShareNotificationMail`を送信。失敗時「メールの送信に失敗しました。しばらく時間をおいて再度お試しください。」、成功時「相手先にメールを送信しました。」

### 5.5 DownloadController（相手先用、認証不要）

認証状態はセッション`download_auth_{token}.{step}`（`passcode_verified`/`email_verified`/`otp_verified`、タイムスタンプとして保持）で管理。①アクセス時に`passcode_verified`は自動付与（旧パスコード画面の自動スキップのため）。

- `showPasscode($token)`：トークン検証→メールアドレス確認画面表示（ルート名は歴史的経緯でpasscodeのまま）
- `verifyEmail($token, Request)`：`RateLimiter`（キー`verify-email:{token}`）で15分間5回まで。6回目以降「試行回数が多すぎます。しばらく時間をおいて再度お試しください。」。入力値が`recipient_email`と不一致の場合「メールアドレスが正しくありません」、`access_logs`に`email_fail`記録。成功時、6桁OTPを`random_int(0,999999)`を0埋め生成、`auth_codes`に新規行作成（既存コードは無効化せず最新行を優先）。`OtpMail`送信失敗時「認証コードの送信に失敗しました。しばらく時間をおいて再度お試しください。」
- `showOtp($token)`：OTP入力画面表示
- `verifyOtp($token, Request)`：`AuthCode`行を`lockForUpdate()`でDBロックし直列化（並列リクエストによる5回ロックアウト回避を防止）。5回連続失敗で`lock_until`を現在時刻+30分にセット。コード比較は`hash_equals`。コード有効期限（10分）切れは「コードの有効期限が切れました。再度URLにアクセスしてください」。不一致は「コードが正しくありません」＋残り試行回数表示。ロック中はロック解除時刻を表示（`download/otp.blade.php`：「認証に5回失敗したためロックされています。**{HH:mm}** 以降に再度お試しください。」）。成功時はダウンロード完了画面（`download.complete`相当）をそのまま応答として表示
- `download($token)`：`DownloadUrl`行を`lockForUpdate()`でDBロックして`download_limit`判定を直列化（並列リクエストによる上限バイパス防止）。ファイルをレスポンスとして返し、`download_count`をインクリメント、`access_logs`に`download`記録。OTP認証完了から10分以内のみ有効（セッション内タイムスタンプ判定、期限切れは①へリダイレクト）。`notify_on_download=true`かつ`download_count`が1になった時（初回のみ）、担当者へ`DownloadNotificationMail`送信（メールボム防止）。送信失敗は握りつぶし（ダウンロード自体は成功扱い）
- 共通チェック（各メソッド冒頭）：トークン不存在→404。`expires_at`過去→`download.error`（`reason: expired`）。`download_limit`到達済み→`download.error`（`reason: limit`）。`deleted_at`あり→`download.error`（`reason: invalid`、存在しないトークンと同じ扱いで情報漏洩防止）

`download/complete.blade.php`の「ダウンロード」ボタンは通常の`<a href>`リンク（AJAXではない）。クリック後、ブラウザは`download.file`への遷移と並行してJS側で400ms後に「ダウンロードが完了しました」という成功メッセージ・ボタン文言「再ダウンロード」への変更・表示中カウンタの+1を行う。**このカウンタ更新は完全にクライアント側の見た目だけの処理**で、実際のサーバー側`download_count`とは同期確認していない（再読み込みすれば実際の値に戻る）。

### 5.6 Admin\UserController

- `index()`：`users`と`depts`をJOINし部署名を含めて表示、最新順、20件ページング
- `store(Request)`バリデーション：
  ```
  name           : required, string, max:255
  employee_code  : nullable, string, max:50
  email          : required, email, unique:users,email
  password       : required, string, min:4, confirmed
  role           : required, in:admin,staff
  is_active      : boolean
  company_id     : nullable, in:P,M,T,H
  dept_id        : nullable, exists:depts,id
  ```
  成功時：「ユーザーを作成しました」→ 一覧へリダイレクト
- `update(Request, User $user)`：同上バリデーションだが`email`は`unique:users,email,{user->id}`（自分は除外）、`password`は`nullable`（未入力なら変更しない）。成功時：「ユーザーを更新しました」
- `destroy(User $user)`：自分自身は削除不可（「自分自身は削除できません」）。他ユーザーは**物理削除**（`users`テーブルに`deleted_at`が無く、`User`モデルも`SoftDeletes`未使用）。`shared_files.user_id`・`download_urls.user_id`・`deleted_files_log.deleted_by`はすべて`users.id`への外部キーで`onDelete('cascade')`設定。成功時：「ユーザーを削除しました」（連鎖削除に関するUI上の追加警告は無い）

### 5.7 Admin\AccessLogController

- `index()`：`AccessLog`を`downloadUrl.sharedFile`・`downloadUrl.user`とともにEager Load、最新順、20件ページング。アクション種別タブ：`all`/`failed`(`email_fail`・`otp_fail`)/`download`(`download`)。クエリパラメータ`status`で絞り込み、クエリ文字列を保持したままページング

### 5.8 Admin\StorageController

- `index()`：全体使用量＝`SharedFile.file_size`合計をMB換算、`config('fileshare.storage_cap_mb')`（10240MB=10GB、暫定）に対する割合（100%でキャップ）。警告しきい値は`storage/app/settings.json`の`storage_warning_threshold`。ユーザー別内訳（user_idでグルーピングした合計サイズ・ファイル数、使用量降順、しきい値以上は「警告」バッジ）。サイズの大きいファイル上位20件
- ユーザー名表示は`optional($row->user)->name ?? '(削除済みユーザー)'`という防御的実装。`shared_files.user_id`が`onDelete('cascade')`のため現状ではユーザー削除時に紐づく`SharedFile`も連鎖削除されこの分岐は実際には到達しないはずだが、孤立データを想定した作りになっている

### 5.9 Admin\SettingController

- `update(Request)`バリデーション：
  ```
  cleanup_grace_days        : required, integer, min:0, max:365
  notify_before_delete      : boolean
  storage_warning_threshold : required, integer, min:1, max:100
  ```
- `storage/app/settings.json`に保存（DBテーブルではない）。**更新時は既存ファイルを読み込んでから上書きする実装にすること**（過去に未読み込みのまま上書きしてデフォルト値で他項目を消してしまった事故があったため）
- デフォルト値：`cleanup_grace_days=7` / `notify_before_delete=false` / `storage_warning_threshold=80`
- 旧「パスコードを必須にする」設定（`passcode_required`）は2026年6月22日に削除済み

### 5.10 自動削除（Artisan Command）

- ファイル：`app/Console/Commands/CleanupExpiredFiles.php`、コマンド名：`fileshare:cleanup`
- 処理フロー：①`expires_at`が猶予日数（デフォルト7日、設定値）以上過去の`download_urls`を取得 → ②削除前通知設定がONの場合、削除予定日7日前に担当者へ`DeleteNotificationMail`送信 → ③猶予期間を過ぎたファイルを`Storage::delete()`で削除（同一ファイルを参照する他のURLがまだ猶予期間内の場合はファイル本体を削除しない） → ④`deleted_files_log`に記録 → ⑤関連する`download_urls`を`forceDelete()`（物理削除）
- 一度もURLに紐づいていないファイルは対象外（`DownloadUrl`を起点に処理するための既知の制限）
- 実行はWindowsタスクスケジューラで1日1回・正午に直接実行（11章参照、Laravelスケジューラの`schedule:run`毎分起動方式は採用していない）

## 8. エラー処理仕様

| 状況 | 処理 |
|------|------|
| トークンが存在しない | `download.error`（`reason: invalid`） |
| URLが期限切れ | `download.error`（`reason: expired`） |
| ダウンロード上限超過 | `download.error`（`reason: limit`） |
| URLが無効化済み（ソフトデリート） | `download.error`（`reason: invalid`、存在しないトークンと区別しない） |
| メールアドレス不一致 | 「メールアドレスが正しくありません」、`access_logs`に`email_fail`記録 |
| メール確認レート制限超過（15分5回） | 「試行回数が多すぎます。しばらく時間をおいて再度お試しください。」 |
| OTP不一致 | 「コードが正しくありません」＋残り試行回数を表示 |
| OTPロック中 | 「認証に5回失敗したためロックされています。**{HH:mm}** 以降に再度お試しください。」 |
| OTP期限切れ | 「コードの有効期限が切れました。再度URLにアクセスしてください」 |
| OTP送信失敗 | 「認証コードの送信に失敗しました。しばらく時間をおいて再度お試しください。」 |
| ログイン失敗 | 「メールアドレスまたはパスワードが正しくありません」 |
| 非アクティブユーザーログイン | 「このアカウントは無効化されています」 |
| URL発行ステップ2でファイル未選択・未アップロード | 「ファイルをアップロードするか、既存のファイルを選択してください。」 |
| 相手先への直接メール送信失敗 | 「メールの送信に失敗しました。しばらく時間をおいて再度お試しください。」 |

`download.error`の`reason`ごとの表示文言（`resources/views/download/error.blade.php`、未知の`reason`は`invalid`と同じ文言にフォールバック）：

| reason | タイトル | 補足文 |
|---|---|---|
| expired | このURLの有効期限は切れています。 | 担当者に新しいURLを発行してもらってください。 |
| limit | ダウンロード可能回数の上限に達しています。 | 担当者にお問い合わせください。 |
| invalid | このURLは無効です。 | URLが正しいかご確認ください。 |

| `DummyDataSeeder` | 担当者2件（`tantosha-a@example.com`／田中一郎、`tantosha-b@example.com`／鈴木花子、いずれも`password='password'`）と、それらに紐づくダミーの`DownloadUrl`10件（取引先・採用・その他の各属性、期限切れ済み・DL上限到達済み・無効化済みなどのバリエーションを含む）。**⚠️ このシーダーは`SharedFile`を新規作成しない。**`original_name`の文字列一致で「事前にアップロード済みのはず」のファイル（`請求書_2026年5月.pdf`等10件、ユーザーA・Bにそれぞれ紐づく）を検索するだけのため、ゼロから`migrate --seed`しただけでは対象ファイルが存在せず`null->id`でFatal Errorになる。再構築時は、シーダー実行前に該当ファイル名で実際にアップロードしておくか、シーダー自体を`SharedFile::create()`も行う自己完結型に書き換えること |

`depts`の初期データは3.2参照（シーダーではなくマイグレーション内で投入）。

### 11.4 サーバー設定

- `public/.htaccess`に`php_value upload_max_filesize 100M`・`php_value post_max_size 110M`をこのプロジェクト限定で設定（XAMPP全体の`php.ini`は変更しない）
- 自動削除（`fileshare:cleanup`）はLaravelスケジューラ（`schedule:run`の毎分起動）を使わず、Windowsタスクスケジューラで1日1回・正午に直接`php artisan fileshare:cleanup`を実行する運用（PCを夜間シャットダウンするため、起動後に自動実行されるよう`-StartWhenAvailable`相当の設定）
- ストレージディスク：Laravelの`local`ディスク。`config/filesystems.php`の`local.root`は`storage_path('app')`（プロジェクト直下）。本番化時にプロジェクトフォルダ外へ分離するかは未決定（要件定義書側の既知の制限・将来課題を参照）
- Apacheリスンポート：8080。VirtualHost設定済み（`C:\xampp\apache\conf\extra\httpd-vhosts.conf`）

### 11.5 バージョン表示機能の実装

- `AppServiceProvider::resolveAppVersion()`が`View::share`経由で**リクエストごとに**`shell_exec()`で`git rev-list --count HEAD`を実行し、`v1.0.{コミット数}`形式のバージョン番号を算出してフッターに表示
- git実行ファイルが見つからない・失敗した場合はバージョン非表示にフォールバック
- 引数はユーザー入力ではなく固定値・`base_path()`のみで`escapeshellarg`済みのため注入リスクは低いが、リクエスト毎にプロセスを起動する設計のため負荷面で要注意（同時アクセス数の目標値確定時には見直し対象）

## 12. 実装上の既知の不整合・技術的負債

- ダッシュボード（`DashboardController::index()`）が`$totalUrls`・`$totalDownloads`・`$validCount`・`$totalSize`を計算しているが、現在のビューはいずれも使用していない（死んでいる変数）
- クリップボードコピー機能が`urls/complete.blade.php`（フォールバック無し）と`urls/show.blade.php`（フィーチャー検出＋フォールバック）の2パターンで混在（7章参照）
- バージョン表示機能がリクエスト毎に`shell_exec()`でgitサブプロセスを起動（11.5参照）。低頻度アクセス前提の現状では問題化していないが本番化時には見直し対象
- ストレージ管理画面（`Admin\StorageController`）が、現状の`onDelete('cascade')`設定下では実際には到達しないはずの「孤立した`shared_files.user_id`」を防御的に想定したコードになっている（5.8参照）
- `FileController::destroy()`の所有者チェックは`DownloadUrlController::authorizeOwner()`と同趣旨だが共通メソッド化されておらずインラインで別実装されている
- `download_urls.passcode`カラムはDBに残存しているが、パスコード認証機能自体は廃止済みで未使用
