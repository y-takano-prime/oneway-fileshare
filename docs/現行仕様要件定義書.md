## 1. システム概要

- **システム名（コード名）:** oneway-fileshare　**UI上のブランド名:** AXON
- **目的:** メール添付によるファイル授受（迷惑メール判定・誤送信・容量制限・履歴管理困難）を解消する、当社→相手先の片方向ファイル授受Webシステム
- **授受の方向:** 当社（担当者）→ 相手先のみ。相手先はダウンロードのみでアップロード機能なし（セキュリティ上の判断により確定。双方向化はスコープ外）
- **ローカルURL:** `http://oneway-fileshare.local:8080`
- **配置場所:** `C:\xampp\htdocs\oneway-fileshare`（2026年6月22日、`C:\xampp\projects\`配下から移行）

## 2. 利用者区分と権限

| 種別 | ログイン | できること | できないこと |
|------|---------|-----------|--------------|
| 管理者（admin） | 必要 | 全URL・全ファイルの閲覧、ユーザー管理、ログ閲覧、ストレージ管理、システム設定変更、全担当者のファイル削除 | **URL発行不可（403）**。アップロードの入口は`urls.store`内（8.2参照）にしかないため、UI非表示と合わせて結果的にアップロードもできない（アップロード単独の403チェックは存在しない） |
| 担当者（staff） | 必要 | 自分が発行したURL・自分がアップロードしたファイルの管理、URL発行、ファイルアップロード・削除 | 他担当者のURLへのID直接指定アクセス（403、`DownloadUrlController::authorizeOwner()`）。ファイルは`FileController::destroy()`内に同趣旨のチェックを別途インラインで実装（共通メソッド化はされていない） |
| 相手先（外部） | 不要（トークンURL） | メールアドレス確認＋OTP認証後のファイルダウンロードのみ | アップロード不可、ログイン不可、他のURLへのアクセス不可 |

管理者にURL発行を禁止しているのは、管理者がメール送信用アカウント（Kagoya SMTP）を持たない運用のため。

## 8. 機能要件（詳細）

業務ルール・ユーザー視点の挙動を記載する。正確なバリデーション文字列・セッションキー名・メソッド名・正確なエラー文言・実装手段は`現行仕様設計仕様書.md`5章を参照。

### 8.1 認証

- メールアドレス＋パスワードでログイン
- 認証失敗時はエラーメッセージを表示
- `is_active=false`（無効化済み）のユーザーはログインできず、専用のエラーメッセージが表示される（2026-06-22修正：以前は専用メッセージが画面に表示されない不整合があったが解消済み。13章参照）
- ログイン成功時はセッションを再生成（セッション固定化対策）
- ログイン済みユーザーが`/login`にアクセスすると自動的にダッシュボードへリダイレクトされる
- ログアウト時はセッション・CSRFトークンを再生成
- 管理者専用機能には管理者以外アクセスできない（403）

### 8.2 ファイルアップロード（担当者のみ）

アップロードの入口は**URL発行フロー内（8.3のステップ2）のみ**。単一ファイルアップロードのみ対応（複数ファイル一括アップロード機能は2026年6月22日に廃止）。

- 許可拡張子・1ファイルあたりの最大サイズ（100MB）は`docs/ファイル制御仕様書.md`参照
- ファイル名は推測困難な値に置き換えて保存する（二重拡張子・なりすまし対策）
- アップロードによりストレージ占有率が警告しきい値を上向きに跨いだ場合、管理者へ警告メールを送信する（8.8参照）

### 8.3 URL発行フロー（3ステップ）

#### ステップ1：送付先設定

| 項目 | 必須 | 補足 |
|---|---|---|
| 属性（取引先／採用／その他） | ○ | 採用選択時は企業名・役職部署欄を自動的に非表示化する |
| 企業名 | △ | 採用選択時は空欄として扱う |
| 役職・部署 | △ | 同上 |
| 相手先の名前 | ○ | |
| 相手先メールアドレス | ○ | |
| 有効期限 | ○ | 未来日時のみ指定可 |
| ダウンロード回数上限 | △ | 未指定の場合は無制限 |
| 備考 | △ | |
| ダウンロード時通知 | - | 相手先がダウンロードした際に担当者へ通知するかどうか |

管理者はこのステップを操作できない（URL発行は担当者のみの機能）。

#### ステップ2：ファイル選択

- アップロード済みファイルから選択するか、新規にファイルをアップロードする（8.2参照）
- ファイル管理画面（8.6）の「URL発行」ボタンから遷移した場合、対象ファイルが事前選択された状態で表示される（2026-06-22修正：以前は機能していなかった不整合があったが解消済み。13章参照）

#### ステップ3：発行完了

- 自分以外の担当者が発行したURLをID指定で開くことはできない（2026-06-22修正：この画面のみ確認が欠落していたセキュリティ上の不整合（IDOR）があったが解消済み。13章参照）
- 送付用のメール文章（宛先・ファイル名・ダウンロードURL・有効期限を含む）が自動生成され、コピーボタンで全文またはURLのみをコピーできる
- 「相手先へ直接メール送信」ボタンから、担当者自身のメールアドレスを送信元として相手先へ直接送信することもできる。送信失敗時・成功時にそれぞれメッセージを表示する

### 8.4 ダウンロード認証フロー（相手先・認証不要）

フロー：①URLアクセス → ②メールアドレス確認 → ③認証コード（OTP）入力画面 → ④OTP確認（成功時はダウンロード完了画面を表示） → ⑤画面内の「ダウンロード」ボタンを押した時のみファイル取得が発生する（自動では発生しない）。

業務ルール：

- メールアドレス確認には試行回数の制限があり、過度な総当たりはブロックされる
- 入力したメールアドレスが発行時の登録アドレスと一致しない場合はエラーメッセージを表示し、ログに記録する
- メールアドレス確認成功時、6桁・有効期限10分のワンタイムコードを登録メールアドレス宛に送信する
- OTP確認は5回連続失敗すると一定時間（30分）ロックされ、ロック中は解除時刻を表示する
- OTPの有効期限切れ・コード不一致はそれぞれ専用のエラーメッセージを表示する（不一致の場合は残り試行回数も表示）
- ファイル取得はOTP認証完了から一定時間（10分）以内のみ有効。それを過ぎた場合は認証フローの最初に戻る
- ダウンロード回数上限に達したURLはダウンロードできない
- 無効化済みのURLと存在しないURLは同じエラー扱いとし、外部から無効化済みかどうかを判別できないようにする（情報漏洩防止）
- 期限切れのURLは専用のエラーを表示する
- ダウンロード成功時はアクセスログに記録し、ダウンロード回数を加算する
- ダウンロード時通知が有効な場合、**初回ダウンロードのみ**担当者へ通知メールを送信する（同一URLで複数回通知しない＝メールボム防止）。通知の送信に失敗してもダウンロード自体は成功扱いとする
- ダウンロード完了画面に表示されるダウンロード回数カウンタは見た目だけの即時反映であり、サーバー側の実際のカウントとは別管理（画面を再読み込みすると実際の値に戻る）

### 8.5 URL管理一覧

- 検索は「相手先（相手先名・相手先メール・ファイル名）」と「担当者名（管理者専用）」の2系統に分かれている（同一人物が相手先・担当者の両方に存在する場合の誤マッチを防ぐため統合しない）
- 状態（未DL／DL済み／期限切れ／無効化済み）・属性（取引先／採用／その他）はそれぞれ複数選択でフィルター可能
- 作成日・有効期限・DL数・相手先・属性でソート可能
- 管理者は全件、担当者は自分が発行した分のみ閲覧可能
- 無効化済みURLも一覧には表示される（アクセス自体は不可だが、一覧からは見える）

### 8.6 ファイル管理

- 担当者は自分のファイルのみ、管理者は全担当者分を閲覧可能
- ファイル名で検索、複数の列でソート可能
- 削除は所有者本人または管理者のみ可能。削除は即時かつ猶予期間なし。紐づくURLも合わせて無効化され、削除履歴ログに記録される
- アップロード機能はこの画面には無い（アップロードはURL発行フローに一本化、8.2参照）
- 各ファイルの「URL発行」ボタンから遷移すると、そのファイルが事前選択された状態でURL発行フローが開始される（2026-06-22修正：以前は機能していなかった不整合があったが解消済み。13章参照）

### 8.7 管理者機能（詳細）

#### 8.7.1 ユーザー管理

- 担当者・管理者アカウントの追加・編集・削除が可能
- 自分自身は削除できない
- **⚠️重要：ユーザー削除はそのユーザーがアップロードした全ファイル・発行した全URL（有効期限内のものを含む）・削除履歴ログまで連鎖して削除する。** 削除前の警告表示は無い。退職者整理等での誤操作により、取引先・応募者向けの有効なダウンロードURLが無警告で消失するリスクがあるため、本番運用前に対応方針を検討すること（12章参照）

#### 8.7.2 ログ管理

- アクセスログを「全件」「失敗のみ」「ダウンロードのみ」で絞り込み表示できる

#### 8.7.3 ストレージ管理

- 全体・ユーザー別のストレージ使用量を確認できる
- 使用率が警告しきい値を超えたユーザーには警告表示が出る

#### 8.7.4 システム設定

- 自動削除までの猶予日数・削除前通知の有無・ストレージ警告しきい値を管理者が設定可能
- 設定値は永続化される（更新時に既存設定の他項目を消さないよう注意すること。2026-06-19に未読み込みのまま上書きしてしまった事故があった経緯がある）
- デフォルト値：猶予日数7日／削除前通知オフ／警告しきい値80%
- 旧「パスコードを必須にする」設定は、パスコード認証機能自体の廃止に伴い2026年6月22日に削除済み

### 8.8 通知メール一覧

| メール | 宛先 | トリガー | 目的 |
|---|---|---|---|
| 認証コード（OTP）メール | 相手先 | メールアドレス確認成功時 | ワンタイムコードを通知 |
| ダウンロード通知メール | 担当者 | 初回ダウンロード時のみ（通知設定がオンの場合） | ダウンロードされたことを通知 |
| 削除前通知メール | 担当者 | 自動削除の7日前（通知設定がオンの場合） | ファイルが削除される旨を通知 |
| ストレージ警告メール | 有効な管理者全員 | ユーザーの占有率が警告しきい値を超えた時 | ストレージ使用状況の警告 |
| 相手先への直接送信メール | 相手先 | 担当者が発行完了画面から送信操作をした時 | URLを相手先へ通知 |

送信元はいずれも担当者自身のメールアドレス（Kagoya SMTP、固定アカウント運用）。各メールの正確な件名・本文・実装クラスは`現行仕様設計仕様書.md`6章参照。

### 8.9 自動削除

- 有効期限切れ後、設定された猶予日数（デフォルト7日）経過したURLとファイルを削除する
- 削除前通知が有効な場合、削除予定日の7日前に担当者へ通知する
- 同一ファイルを参照する他のURLがまだ猶予期間内の場合、ファイル本体は削除しない
- 一度もURLに紐づいていないファイルは自動削除の対象外（既知の制限）
- 削除履歴ログは1年経過後に自動削除される（詳細は`docs/データ保持ルール.md`）
- 実行は1日1回・正午（実装詳細は`現行仕様設計仕様書.md`5.10参照）

### 8.10 ダッシュボード

- 発行URLの状況をメトリクスカード（DL済み・未DL・期限切れ・無効化済み・ストレージ使用量）で表示。各カードはクリックでURL管理一覧の該当フィルターに遷移する
- ストレージ使用量が警告しきい値以上の場合、**担当者にのみ**警告を表示する（管理者には表示しない）
- 直近のURL一覧を表示する（管理者の場合のみ担当者列も表示）
- 担当者には「＋新規作成」ボタンを表示する（管理者はURL発行不可のため非表示）

### 8.11 URL詳細

- 自分以外の担当者が発行したURLをID指定で開くことはできない（管理者は閲覧可）
- ファイル名・相手先情報・有効期限・DL数・ダウンロードURL・備考を表示する
- 無効化済みの場合は編集・無効化操作ができないようにする（再度の操作を防ぐ）
- 送付メール文章を再表示・コピー可能（このURL詳細画面に再送信フォームは無い＝相手先への再送信は発行完了画面限定）
- アクセスログを確認できる
- 「無効化」操作には確認ダイアログを出す

### 11.1 セキュリティ（指摘3対応）

| 項目 | 内容 |
|---|---|
| パスワード | ハッシュ化して保存（平文保存はしない） |
| ファイル保存場所 | Web非公開のディスクに保存し、認証フローを経由しない限り取得不可 |
| HTTPセキュリティヘッダー | 全ページにクリックジャッキング対策・MIME種別推測防止・リファラポリシー・HSTSを付与。CSPは未導入（インラインstyle多用のため見送り、別途対応が必要） |
| IDOR対策 | 管理者以外が他人のURLにID直接指定でアクセスした場合は403とする。URL関連の全操作に適用（2026-06-22に1箇所欠落が発覚し修正済み、13章参照） |
| トークン | URL発行時に推測困難な長さ（64文字）のランダム文字列を生成 |
| レート制限 | メールアドレス確認：15分間5回まで。OTP：5回失敗で30分ロック |
| 競合状態対策 | OTPロック判定・ダウンロード回数判定は並列リクエストによるバイパスを防止する実装にする |
| CSRF | 全フォームにCSRF保護を適用 |
| 管理者の操作制限 | URL発行を禁止（結果的にアップロードも行えない、2章参照） |
| シェルコマンド実行 | バージョン表示機能（11.4参照）がgitコマンドを実行する。固定値のみを引数とするため注入リスクは低いが、リクエスト毎にプロセスを起動する設計である点は負荷面で要注意（11.5参照） |

実装詳細（ヘッダーの正確な値、`authorizeOwner()`の適用範囲、`lockForUpdate`等の実装手段）は`現行仕様設計仕様書.md`5.1・5.4・5.5・2章参照。

### 11.2 ファイル制御（指摘4対応）

詳細は`docs/ファイル制御仕様書.md`を参照。要点：

- 許可拡張子：jpg/jpeg/png/gif/pdf/doc/docx/xls/xlsx/ppt/pptx/zip/csv/txt
- 1ファイル最大100MB
- 空ファイル（0バイト）の明示的な拒否は**未対応**（既知の制限）
- 二重拡張子対策はLaravelの拡張子判定ルール（実体バイト列からの推定）に依存、専用テストは無し

### 11.3 品質保証・テスト（指摘2対応）

詳細は`docs/テスト仕様書.md`を参照。自動テストは本書作成時点で0件。同文書のテスト観点を手動確認チェックリストとして運用する。

### 11.4 バージョン管理（指摘5対応）

- Gitのコミット数を起点にバージョン番号を算出し、全ページのフッターに表示する
- コミットするたびに番号が自動的に増加するため、手動でのバージョン管理は不要
- リモート：GitHub（`https://github.com/y-takano-prime/oneway-fileshare.git`）。`main`ブランチで運用
- 実装方式（リクエスト毎の`shell_exec()`実行、git実行不可時のフォールバック）は`現行仕様設計仕様書.md`11.5参照

### 11.5 容量制限・負荷対策（指摘6対応）

- 1ファイル最大100MB（8.2参照）。アップロードは単一ファイルのみのため「1回のアップロード合計」という概念は本実装では発生しない（複数ファイル一括アップロード機能は廃止済み）
- ユーザー単位のストレージ総量上限：現在10GB（暫定）。超過しても警告のみでアップロード自体は継続
- 想定同時アクセス数：明確な目標値は未設定（社内利用＋取引先・応募者からの低頻度アクセスを想定。プロトタイプ運用のため数値目標は今後の本番化検討時に確定予定）
- 既知の負荷要因：バージョン表示機能（11.4）が全リクエストでgitサブプロセスを起動している。低頻度アクセス前提の現状では問題化していないが、本番化・同時アクセス数の目標値確定時には見直し対象とする

## 12. 既知の制限・将来課題

- ウイルスチェック未実装（本番リリース前必須、ツール未確定）
- ファイルアップロード先は、クラウドストレージ（S3/R2）案ではなく**社内システムサーバーに直接アップロードする方針に変更**（2026年6月22日時点）。現状の保存先は`config/filesystems.php`の`local`ディスク＝`storage_path('app')`（プロジェクトフォルダ内）のままで問題ないか、コードのデプロイ・フォルダ移動からデータを分離するためプロジェクトフォルダ外の別パスに切り出すべきか**未決定（予定）**。`Storage::`ファサード経由の実装のため、変更時もコントローラー側の修正は不要（`root`設定のみ変更で対応可能）
- 双方向ファイル授受は対応しない（セキュリティ上の判断で確定）
- 自動テスト0件
- 空ファイルアップロードの明示的拒否なし
- 一度もURLに紐づいていないファイルは自動削除対象外
- CSP未導入
- **ユーザー削除時の連鎖削除リスク（要再検討）：** `admin.users.destroy`はユーザーを物理削除し、外部キーの`onDelete('cascade')`設定により、そのユーザーの`shared_files`・`download_urls`（有効期限内のものを含む）・`deleted_files_log`まで連鎖削除される。確認ダイアログ等の警告は無い。退職者整理等での誤操作で取引先・応募者向けの有効なURLが無警告で消える可能性があるため、本番運用前に「ソフトデリート化」「削除前の影響確認画面」「ユーザー無効化（`is_active=false`）への一本化で物理削除自体を提供しない」等の対応方針を検討すること（8.7.1参照）
