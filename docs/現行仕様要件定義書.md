# 現行仕様要件定義書（実装反映版・ゼロ再構築用）

## 0. 本書の位置づけ

`docs/要件定義書.md`（初版、2026年6月15日作成・読み取り専用）に対して、打合せで以下6点の指摘を受けた。

1. 仕様の粒度が荒い（機能名レベルの記述が多く、具体的な動作・エラー処理・UI文言が不明）
2. 受け入れ条件・テスト観点がなく、何をもって完成かの基準がない
3. セキュリティ要件が浅い（ハッシュ化・ファイル保存場所・HTTPヘッダー等の詳細が欠如）
4. ファイル制御が未定義（許可拡張子・空ファイル・二重拡張子）
5. バージョン管理がなく、変更履歴が追えない
6. 1案件の合計容量がなく、大量アップロード時の負荷対策が不明

初版`要件定義書.md`は読み取り専用のため上書きせず、**現在の実装を正として本書を新規作成**した。本書は実装の変更に追従して更新していく前提の文書である。

**本書の到達目標（2026年6月22日改訂）：** 本書単体（＋本章の関連文書）だけを渡された、このリポジトリのソースコードに一切アクセスできない別のAIエージェント（想定：Codex GPT PRO環境）が、ゼロから現在と機能的に同等なシステムを再構築できるレベルの情報量を持つこと。デザイン（CSS/HTMLの厳密な再現）より、**動作・データ構造・セキュリティ要件の完全一致を優先する**。

## 1. システム概要

- **システム名（コード名）:** oneway-fileshare　**UI上のブランド名:** AXON
- **目的:** メール添付によるファイル授受（迷惑メール判定・誤送信・容量制限・履歴管理困難）を解消する、当社→相手先の片方向ファイル授受Webシステム
- **授受の方向:** 当社（担当者）→ 相手先のみ。相手先はダウンロードのみでアップロード機能なし（セキュリティ上の判断により確定。双方向化はスコープ外）
- **ローカルURL:** `http://oneway-fileshare.local:8080`
- **配置場所:** `C:\xampp\htdocs\oneway-fileshare`（2026年6月22日、`C:\xampp\projects\`配下から移行）

## 2. 利用者区分と権限

| 種別 | ログイン | できること | できないこと |
|------|---------|-----------|--------------|
| 管理者（admin） | 必要 | 全URL・全ファイルの閲覧、ユーザー管理、ログ閲覧、ストレージ管理、システム設定変更、全担当者のファイル削除 | **URL発行不可（403）**。アップロードの入口は`urls.store`内（8.2参照）にしかないため、UI非表示と合わせて結果的にアップロードもできない（アップロード単独の403チェックは存在しない） |
| 担当者（staff） | 必要 | 自分が発行したURL・自分がアップロードしたファイルの管理、URL発行、ファイルアップロード・削除 | 他担当者のURLへのID直接指定アクセス（403、`DownloadUrlController::authorizeOwner()`）。ファイルは`FileController::destroy()`内に同趣旨のチェックを別途インラインで実装（共通メソッド化はされていない） |
| 相手先（外部） | 不要（トークンURL） | メールアドレス確認＋OTP認証後のファイルダウンロードのみ | アップロード不可、ログイン不可、他のURLへのアクセス不可 |

管理者にURL発行を禁止しているのは、管理者がメール送信用アカウント（Kagoya SMTP）を持たない運用のため。

### 3.4 サーバー設定

- `public/.htaccess`に`php_value upload_max_filesize 100M`・`php_value post_max_size 110M`をこのプロジェクト限定で設定（XAMPP全体の`php.ini`は変更しない）
- 自動削除（`fileshare:cleanup`）はLaravelスケジューラ（`schedule:run`の毎分起動）を使わず、Windowsタスクスケジューラで1日1回・正午に直接`php artisan fileshare:cleanup`を実行する運用（PCを夜間シャットダウンするため、起動後に自動実行されるよう`-StartWhenAvailable`相当の設定）
- ストレージディスク：Laravelの`local`ディスク。`config/filesystems.php`の`local.root`は`storage_path('app')`（プロジェクト直下）。本番化時にプロジェクトフォルダ外へ分離するかは未決定（12章参照）

## 5. モデル仕様

| モデル | $fillable | $casts | リレーション | 独自メソッド／スコープ |

## 8. 機能要件（詳細）

### 8.1 認証

- メールアドレス＋パスワードでログイン（Laravel標準のbcryptハッシュ照合、`Auth::attempt()`）
- バリデーション：`email`(required, email, max:255) / `password`(required, string, min:4)
- 認証失敗時：コントローラーは`withErrors(['email' => 'メールアドレスまたはパスワードが正しくありません'])`を設定（入力欄はemailのみ復元）
- `is_active=false`のユーザーは認証成功後に即ログアウトされ、コントローラーは`withErrors(['email' => 'このアカウントは無効化されています'])`を設定する
- ビュー（`auth/login.blade.php`）は`{{ $errors->first('email') }}`でこのメッセージをそのまま表示する（2026-06-22修正：以前は固定文字列を表示するだけで「このアカウントは無効化されています」が画面に出ない不整合があったが解消済み。13章参照）
- ログイン成功時：セッション再生成（セッション固定化対策）→ `intended()`または`/dashboard`へリダイレクト
- ログイン済みユーザーが`/login`にアクセスすると`/dashboard`へリダイレクト（`guest`ミドルウェア）
- ログアウト：セッション無効化＋CSRFトークン再生成
- 管理者用ミドルウェア（`AdminMiddleware`）：未認証または`role !== 'admin'`の場合403

### 8.2 ファイルアップロード（担当者のみ）

アップロードの入口は**URL発行フロー内（8.3のステップ2）のみ**。単一ファイルアップロード（`upload_file`フィールド）で、複数ファイル一括アップロード機能は2026年6月22日に廃止（旧`ファイル管理`画面のドロップエリア・`FileController::store()`を削除）。

- バリデーション：`required, file, max:102400(=100MB), mimes:jpg,jpeg,png,gif,pdf,doc,docx,xls,xlsx,ppt,pptx,zip,csv,txt`
- 保存先：`storage/app/uploads/{user_id}/{UUID}.{拡張子}`（`local`ディスク、Web非公開）。ファイル名はUUIDに完全置換（二重拡張子・なりすまし対策、4章参照）
- アップロード前後のファイル合計サイズを比較し、ストレージ警告しきい値を上向きに跨いだ場合は管理者へ`StorageWarningMail`を送信（8.8参照）
- 詳細は`docs/ファイル制御仕様書.md`参照

### 8.3 URL発行フロー（3ステップ）

#### ステップ1：送付先設定（`urls.create` → `urls.store_step1`）

| 項目 | 必須 | バリデーション | 補足 |
|---|---|---|---|
| 属性（category） | ○ | `required, in:business,recruitment,other` | 取引先／採用／その他。採用選択時は企業名・役職部署欄をJSで非表示化し送信時に空にする |
| 企業名 | △ | `nullable, string, max:255` | |
| 役職・部署 | △ | `nullable, string, max:255` | |
| 相手先の名前 | ○ | `required, string, max:255` | |
| 相手先メールアドレス | ○ | `required, email, max:255` | |
| 有効期限 | ○ | `required, date, after:now` | |
| ダウンロード回数上限 | △ | `nullable, integer, min:1, max:9999` | |
| 備考 | △ | `nullable, string, max:2000` | |
| ダウンロード時通知 | - | boolean（`$request->boolean()`で変換） | |

入力値はDBに保存せず、セッション（`create_recipient`）に一時保持してステップ2へ。管理者がこのアクションを呼ぶと403「URL発行は担当者のみ操作できます」。

#### ステップ2：ファイル選択（`urls.create_step2` → `urls.store`）

- セッションに`create_recipient`が無ければステップ1へリダイレクト
- 表示内容：自分（`Auth::id()`）がアップロード済みの`SharedFile`一覧（`latest()`順）
- 「アップロード済みから選択」：`shared_file_id`を送信。バリデーション：`required`, `Rule::exists('shared_files','id')->where('user_id', Auth::id())`（他人のファイルIDは拒否）。エラー文言：「ファイルをアップロードするか、既存のファイルを選択してください。」
- 「新規アップロード」：`upload_file`を送信（8.2のバリデーション）。アップロード成功時、即座に`category=null`の`SharedFile`を作成
- いずれか成立後、`DownloadUrl`を作成：`token`は`Str::random(64)`、`download_count`は0で初期化、ステップ1のセッション値とマージして保存。保存後セッションの`create_recipient`を破棄し、`urls.complete`へリダイレクト

#### ステップ3：発行完了（`urls.complete`）

- **`authorizeOwner()`必須**（2026-06-22修正：他人のURLを直接ID指定で開けないようにするセキュリティ修正済み。13章参照）
- 自動生成される送付メール文章（`buildMailBody()`）の構造：
  ```
  （企業名）
  （役職・部署）
  （企業名または役職・部署があれば空行）
  （相手先名） 様

  お世話になっております。
  以下のURLよりファイルをダウンロードいただけます。

  ■ファイル名
  （ファイル名）

  ■ダウンロードURL
  （download.passcodeルートの完全URL、トークン付き）

  ■有効期限
  （Y年m月d日 H:i形式）

  ダウンロード後は手順に従いメール認証を完了してください。

  よろしくお願いいたします。
  ```
- 「全文をコピー」「DL URLのコピー」ボタン：実装は`navigator.clipboard`を一切呼ばず、**常に**`textarea`を選択して`document.execCommand('copy')`を実行するのみ（HTTP環境では`navigator.clipboard`自体が使えないため、フォールバック分岐ではなく最初からこの方式に統一している）。クリック後は2秒間ボタン文言が「コピーしました！」/「コピー済み」に変化する
- 「相手先へ直接メール送信」（`urls.send_mail`、`authorizeOwner()`必須）：担当者自身のメールアドレスを送信元として`FileShareNotificationMail`（件名固定「ファイルのご案内」、本文は上記`buildMailBody()`そのまま）を相手先に送信。送信失敗時は「メールの送信に失敗しました。しばらく時間をおいて再度お試しください。」、成功時は「相手先にメールを送信しました。」

### 8.4 ダウンロード認証フロー（相手先・認証不要）

```
① GET  /d/{token}/        → メールアドレス確認画面（旧パスコード画面を自動スキップ）
② POST /d/{token}/email   → メールアドレス一致判定
③ GET  /d/{token}/otp     → 認証コード入力画面
④ POST /d/{token}/otp     → コード判定 → 成功時はダウンロード完了画面（download.complete）をそのまま応答として表示
⑤ GET  /d/{token}/download → ④の画面内「ダウンロード」ボタンをユーザーが押した時のみ発生（自動では発生しない）。route名`download.file`、ファイル取得
```

| ステップ | 制限・条件 | 失敗時の挙動 |
|---|---|---|
| ②メール確認 | 同一トークンに対し15分間に5回まで（`RateLimiter`、キー`verify-email:{token}`） | 6回目以降「試行回数が多すぎます。しばらく時間をおいて再度お試しください。」 |
| ②メール確認 | 入力値が`recipient_email`と不一致 | 「メールアドレスが正しくありません」、`access_logs`に`email_fail`記録 |
| ②成功時 | 6桁OTPを生成（`random_int(0,999999)`を0埋め）、有効期限10分。`auth_codes`に新規行を作成（既存コードは無効化されず単に最新行が優先される） | `OtpMail`送信失敗時は「認証コードの送信に失敗しました。しばらく時間をおいて再度お試しください。」 |
| ④OTP確認 | 5回連続失敗で30分ロック（`AuthCode`行をDBロック（`lockForUpdate`）で直列化し、並列リクエストによる5回ロックアウト回避を防止） | ロック中はロック解除時刻を表示 |
| ④OTP確認 | コード有効期限（10分）切れ | 「コードの有効期限が切れました。再度URLにアクセスしてください」 |
| ④OTP確認 | コード不一致（`hash_equals`で比較） | 「コードが正しくありません」＋残り試行回数を表示 |
| ⑤ダウンロード | OTP認証完了から10分以内のみ有効（セッション内タイムスタンプで判定） | 期限切れは認証フローの最初（①）へリダイレクト |
| ⑤ダウンロード | `download_limit`到達済み（`DownloadUrl`行をDBロックで判定→加算を直列化し、並列リクエストによる上限バイパスを防止） | `download.error`（`reason: limit`） |
| 全ステップ共通 | URLが無効化済み（ソフトデリート）・存在しない | `download.error`（`reason: invalid`）、無効化済みと存在しないトークンを区別しない（情報漏洩防止） |
| 全ステップ共通 | URLが期限切れ | `download.error`（`reason: expired`） |

`download.error`の`reason`ごとの表示文言（`resources/views/download/error.blade.php`、未知の`reason`は`invalid`と同じ文言にフォールバック）：

| reason | タイトル | 補足文 |
|---|---|---|
| expired | このURLの有効期限は切れています。 | 担当者に新しいURLを発行してもらってください。 |
| limit | ダウンロード可能回数の上限に達しています。 | 担当者にお問い合わせください。 |
| invalid | このURLは無効です。 | URLが正しいかご確認ください。 |

OTPロック中（`download/otp.blade.php`）の表示文言：「認証に5回失敗したためロックされています。**{ロック解除時刻のHH:mm}** 以降に再度お試しください。」

ダウンロード完了画面（`download/complete.blade.php`）の「ダウンロード」ボタンは通常の`<a href>`リンク（AJAXではない）。クリック後、ブラウザは`download.file`への遷移＝ファイル取得を行うのと並行して、JS側で400ms後に「ダウンロードが完了しました」という成功メッセージとボタン文言の「再ダウンロード」への変更、表示中のダウンロード回数カウンタの+1を行う。**このカウンタ更新は完全にクライアント側の見た目だけの処理**で、実際のサーバー側`download_count`とは同期確認していない（再読み込みすれば実際の値に戻る）。

- 各ステップの認証状態は**セッション**（`download_auth_{token}.{step}`、stepは`passcode_verified`/`email_verified`/`otp_verified`）にタイムスタンプとして保持。①アクセス時に`passcode_verified`は自動付与（旧パスコード画面の自動スキップのため）
- ダウンロード成功時：`access_logs`に`download`記録、`download_count`をインクリメント
- `notify_on_download=true`の場合、**初回ダウンロードのみ**（`download_count`が1になった時）担当者へ`DownloadNotificationMail`を送信（メールボム防止）。送信失敗は握りつぶし（ダウンロード自体は成功扱い）

### 8.5 URL管理一覧（`urls.index`）

- 検索：「相手先（`q`：相手先名・相手先メール・ファイル名のいずれかにLIKE一致）」と「担当者名（`staff_q`、管理者専用、全角/半角スペース無視）」の2系統。`REPLACE(REPLACE(name, ' ', ''), '　', '') LIKE ?`でスペース有無を無視
- 状態フィルター（複数選択、OR結合）：`wait`(未DL=期限内かつdownload_count=0) / `done`(DL済み=期限内かつdownload_count>0) / `expired`(期限切れ) / `invalidated`(ソフトデリート済み)。未選択時はフィルターなし＝全件表示（無効化済みも含む）
- 属性フィルター（複数選択、OR結合）：`business` / `recruitment` / `other`
- 各タブの件数バッジは検索条件のみ反映し、状態・属性フィルター適用前の数を表示
- ソート可能列：`created_at`(作成日) / `expires_at`(有効期限) / `download_count`(DL数) / `recipient_name`(相手先) / `category`(属性)。クリックで昇順/降順切替、↑↓表示。ページサイズ20件
- 管理者は全件、担当者は自分が発行した分のみ（`user_id`で絞り込み）
- 無効化済みURLも一覧に表示される（404にはしない、`withTrashed()`）

### 8.6 ファイル管理（`files.index`）

- 担当者は自分のファイルのみ、管理者は全担当者分を閲覧
- 検索：ファイル名（`q`）
- ソート可能列：ファイル名・URL数・アップロード者（管理者のみ表示）・サイズ・アップロード日時・最短有効期限（`earliest_expires_at`＝紐づくURLのうち最も早い`expires_at`。画面上は「削除予定日」と表示されるが、これは表示時に猶予日数を加算して算出した値であり、ソートキー自体は猶予日数を加味していない）
- 削除（`files.destroy`）：所有者本人または管理者のみ可（インラインチェック、`authorizeOwner()`とは別実装）。即時・物理削除（猶予期間なし）。紐づく全`DownloadUrl`をソフトデリート。`deleted_files_log`に記録
- **アップロードUIは無し**（2026年6月22日に撤去。アップロードは8.3ステップ2に一本化）
- 各ファイル行の「URL発行」ボタン（担当者のみ表示）は`route('urls.create', ['shared_file_id' => $file->id])`、つまり`urls.create`に`shared_file_id`をクエリパラメータとして付与してリンクしている。`DownloadUrlController::create()`がこの値を自分のファイルかどうか確認の上で受け取り、ステップ1の隠しフィールド→セッション（`create_preselected_file_id`）→ステップ2のファイル選択ラジオボタンの事前チェックまで引き継がれる（2026-06-22修正：以前はこのクエリパラメータが一切使われず事前選択が機能していなかった不整合があったが解消済み。13章参照）

### 8.7 管理者機能（詳細）

#### 8.7.1 ユーザー管理

- 一覧（`admin.users.index`）：`users`と`depts`をJOINし部署名を含めて表示、最新順、20件ページング
- 新規作成（`admin.users.create`→`store`）バリデーション：
  ```
  name           : required, string, max:255
  employee_code  : nullable, string, max:50
  email          : required, email, unique:users,email
  password       : required, string, min:4, confirmed
  role           : required, in:admin,staff
  is_active      : boolean
  company_id     : nullable, in:P,M,T,H
  dept_id        : nullable, exists:depts,id
  ```
  成功時：「ユーザーを作成しました」→ 一覧へリダイレクト
- 編集（`admin.users.edit`→`update`）：同上バリデーションだが`email`は`unique:users,email,{自分のid}`（自分は除外）、`password`は`nullable`（未入力なら変更しない）
  成功時：「ユーザーを更新しました」
- 削除（`admin.users.destroy`）：自分自身は削除不可（「自分自身は削除できません」）。他ユーザーは**物理削除**（`users`テーブルに`deleted_at`が無く、`User`モデルも`SoftDeletes`を使用していないため、ソフトデリートではない）。成功時：「ユーザーを削除しました」
  - **⚠️ 重要：連鎖削除の影響範囲。** `shared_files.user_id`・`download_urls.user_id`・`deleted_files_log.deleted_by`はすべて`users.id`への外部キーで`onDelete('cascade')`が設定されている。そのためユーザーを1人削除すると、そのユーザーが**アップロードした全ファイル・発行した全URL（有効期限内で相手先が現在使用中のものを含む）・削除履歴ログまでDBレベルで連鎖削除される**。UIには「ユーザーを削除しました」以外の警告は無い。退職者整理等での誤操作により、取引先・応募者向けの有効なダウンロードURLが無警告で消失するリスクがある（12章にも記載）

#### 8.7.2 ログ管理（`admin.logs.index`）

- `AccessLog`を`downloadUrl.sharedFile`・`downloadUrl.user`とともにEager Load、最新順、20件ページング
- アクション種別タブ：`all`(全件) / `failed`(`email_fail`・`otp_fail`) / `download`(`download`)。各タブの件数も表示
- クエリパラメータ`status`で絞り込み、クエリ文字列を保持したままページング

#### 8.7.3 ストレージ管理（`admin.storage.index`）

- 全体使用量：`SharedFile.file_size`の合計をMB換算、`config('fileshare.storage_cap_mb')`（10240MB=10GB、暫定）に対する割合（100%でキャップ）
- 警告しきい値：`storage/app/settings.json`の`storage_warning_threshold`（デフォルト80%）
- ユーザー別内訳：user_idでグルーピングした合計サイズ・ファイル数、使用量降順。各行の使用率が警告しきい値以上の場合「警告」バッジを表示
- サイズの大きいファイル上位20件（ユーザー情報付き）
- ユーザー名表示は`optional($row->user)->name ?? '(削除済みユーザー)'`という防御的な実装になっている。`shared_files.user_id`は`onDelete('cascade')`のため、現状の仕組みではユーザー削除時に紐づく`SharedFile`も連鎖削除され、この分岐は実際には到達しないはずだが、コード上は孤立データを想定した作りになっている

#### 8.7.4 システム設定（`admin.settings.index`→`update`）

バリデーション：
```
cleanup_grace_days        : required, integer, min:0, max:365
notify_before_delete      : boolean
storage_warning_threshold : required, integer, min:1, max:100
```
- `storage/app/settings.json`に保存（DBテーブルではない）。**更新時は既存ファイルを読み込んでから上書きする**（過去に未読み込みのまま上書きしてデフォルト値で他項目を消してしまった事故があったため）
- デフォルト値：`cleanup_grace_days=7` / `notify_before_delete=false` / `storage_warning_threshold=80`
- 旧「パスコードを必須にする」設定（`passcode_required`）は、パスコード認証機能自体が廃止済みのため2026年6月22日に削除

### 8.8 通知メール一覧（詳細）

### 8.9 自動削除（`fileshare:cleanup`、Windowsタスクスケジューラで1日1回・正午実行）

- 有効期限切れ後、設定された猶予日数（デフォルト7日）経過したURLを物理削除（`forceDelete`）
- 同一ファイルを参照する他のURLがまだ猶予期間内の場合、ファイル本体は削除しない
- 一度もURLに紐づいていないファイルは対象外（既知の制限。`fileshare:cleanup`は`DownloadUrl`を起点に処理するため）
- `deleted_files_log`は1年経過後に自動削除

### 8.10 ダッシュボード（`dashboard`）

- 表示対象は`DownloadUrl`：管理者は全件、担当者は自分が発行した分のみ（`user_id`で絞り込み）
- メトリクスカード5枚（いずれもクリックでURL管理一覧の該当フィルターへ遷移）：
  1. **DL済み**：`expires_at`が未来かつ`download_count > 0`の件数
  2. **未DL**：`expires_at`が未来かつ`download_count = 0`の件数
  3. **期限切れ**：`expires_at`が過去の件数
  4. **無効化済み**：ソフトデリート済み（`onlyTrashed()`）の件数
  5. **ストレージ**：使用量MB／上限GB表示＋割合バー。アップロード済みファイル件数も表示。クリック先は管理者なら`admin.storage.index`、担当者なら`files.index`
- ストレージ使用量が警告しきい値（`storage_warning_threshold`、デフォルト80%）以上の場合、**担当者にのみ**警告アラートを表示（「ストレージ使用量が全体容量の{n}%に達しています…」）。管理者には表示しない
- 「直近のURL」テーブル：最新10件（`recentUrls`）。列は相手先・企業・役職部署・メールアドレス・属性バッジ・ファイル名（リンク）・作成日・有効期限・DL数／上限・状態バッジ。管理者の場合のみ先頭に「担当者」列が付く
- 担当者には画面右上に「＋新規作成」ボタンを表示（管理者には非表示。管理者はURL発行不可のため）
- **注意：** `DashboardController::index()`は`$totalUrls`・`$totalDownloads`・`$validCount`・`$totalSize`も計算しているが、現在のビューはいずれも使用していない（死んでいる変数）。再構築時はビューで使う値のみ計算すればよい

### 8.11 URL詳細（`urls.show`）

- `authorizeOwner()`必須（管理者以外は他人のURLを直接ID指定で開けない）
- 表示内容：ファイル名・（あれば）企業名・相手先名＋役職部署＋メールアドレス・有効期限（期限切れなら`badge-expired`を併記）・DL数／上限・ダウンロードURL（コピー可）・（あれば）備考
- 無効化済み（ソフトデリート済み）の場合、見出しに「無効化済み」バッジを表示し、「編集」「無効化」ボタン自体を非表示にする（再度の無効化や編集を防ぐ）
- 送付メール文章を`buildMailText()`で再生成して表示・コピー可能（8.3ステップ3と同じ内容を後からも確認できる。このURL詳細画面に再送信フォームは無い＝再送信は発行完了画面限定）
- アクセスログ一覧（`access_logs`を最新順、20件ページング）：日時・IPアドレス・アクション（日本語ラベル）
- 「URLを無効化」ボタンはクリック時に確認ダイアログ（`confirm('このURLを無効化しますか？')`）を出す。「有効期限・上限を編集」ボタンは`urls.edit`へのリンク
- コピー機能はこの画面だけ`navigator.clipboard`によるフィーチャー検出＋フォールバックの実装（9章参照）

## 11. 非機能要件（詳細）

### 11.1 セキュリティ（指摘3対応）

| 項目 | 内容 |
|---|---|
| パスワード | Laravel標準bcryptでハッシュ化 |
| ファイル保存場所 | `storage/app/uploads/`（`local`ディスク）。`public`ディスクではなくシンボリックリンクも無いため、URL直接アクセス不可。`DownloadController`の認証フローを経由しない限り取得不可 |
| HTTPセキュリティヘッダー | `app/Http/Middleware/SecurityHeaders.php`で全ページに付与：`X-Frame-Options: SAMEORIGIN`、`X-Content-Type-Options: nosniff`、`Referrer-Policy: strict-origin-when-cross-origin`、`Strict-Transport-Security: max-age=31536000; includeSubDomains`。CSPは未導入（インラインstyle多用のため見送り、別途対応が必要） |
| IDOR対策 | `DownloadUrlController::authorizeOwner()`で、管理者以外が他人のURLにID直接指定でアクセスした場合403。`show`/`edit`/`update`/`destroy`/`sendMail`/`complete`の全アクションに適用（`complete`は2026-06-22に欠落が発覚し修正済み、13章参照） |
| トークン | URL発行時に64文字のランダム文字列を生成（`Str::random(64)`） |
| レート制限 | メールアドレス確認：15分間5回。OTP：5回失敗で30分ロック |
| 競合状態対策 | OTPロック判定・ダウンロード回数判定はDB行ロック（`lockForUpdate`）でトランザクション内に直列化し、並列リクエストによるバイパスを防止 |
| CSRF | Laravel標準のCSRF保護を全フォームに適用（除外設定なし） |
| 管理者の操作制限 | URL発行を403で禁止（コントローラー・ビュー両方）。アップロードの入口がURL発行フロー内にしかないため、結果的にアップロードも行えない（2章参照） |
| シェルコマンド実行 | バージョン表示機能（11.4参照）が`shell_exec()`でgitコマンドを実行。引数はユーザー入力ではなく固定値・`base_path()`のみで`escapeshellarg`済みのため注入リスクは低いが、リクエスト毎にプロセスを起動する設計である点は要注意（負荷面は11.5参照） |

### 11.2 ファイル制御（指摘4対応）

詳細は`docs/ファイル制御仕様書.md`を参照。要点：

- 許可拡張子：jpg/jpeg/png/gif/pdf/doc/docx/xls/xlsx/ppt/pptx/zip/csv/txt
- 1ファイル最大100MB（PHPサーバー側`upload_max_filesize=100M`、`public/.htaccess`でこのプロジェクトに限定して設定）
- 空ファイル（0バイト）の明示的な拒否は**未対応**（既知の制限）
- 二重拡張子対策はLaravelの`mimes`ルール（実体バイト列からの推定）に依存、専用テストは無し

### 11.3 品質保証・テスト（指摘2対応）

詳細は`docs/テスト仕様書.md`を参照。自動テストは本書作成時点で0件。同文書のテスト観点を手動確認チェックリストとして運用する。

### 11.4 バージョン管理（指摘5対応）

- Gitのコミット数を起点に`v1.0.{コミット数}`形式のバージョン番号を算出し、全ページのフッターに表示（`AppServiceProvider::resolveAppVersion()`）
- コミットするたびに番号が自動的に増加するため、手動でのバージョン管理は不要
- リモート：GitHub（`https://github.com/y-takano-prime/oneway-fileshare.git`）。`main`ブランチで運用
- **実装上の注意：** バージョン番号は`View::share`経由で**リクエストごとに**`shell_exec()`で`git rev-list --count HEAD`を実行して算出している（キャッシュなし）。git実行ファイルが見つからない・失敗した場合はバージョン非表示にフォールバックする。リクエスト毎にプロセスを起動するため、11.5の負荷想定が確定する際はこの処理コストも合わせて見直すこと

### 11.5 容量制限・負荷対策（指摘6対応）

- 1ファイル最大100MB（8.2参照）。アップロードは単一ファイルのみのため「1回のアップロード合計」という概念は本実装では発生しない（複数ファイル一括アップロード機能は廃止済み）
- ユーザー単位のストレージ総量上限：`config/fileshare.php`の`storage_cap_mb`（現在10GB、暫定）。超過しても警告のみでアップロード自体は継続
- 想定同時アクセス数：明確な目標値は未設定（社内利用＋取引先・応募者からの低頻度アクセスを想定。プロトタイプ運用のため数値目標は今後の本番化検討時に確定予定）
- 既知の負荷要因：11.4記載のバージョン表示機能が全リクエストでgitサブプロセスを起動している。低頻度アクセス前提の現状では問題化していないが、本番化・同時アクセス数の目標値確定時には見直し対象とする

## 12. 既知の制限・将来課題

- ウイルスチェック未実装（本番リリース前必須、ツール未確定）
- ファイルアップロード先は、クラウドストレージ（S3/R2）案ではなく**社内システムサーバーに直接アップロードする方針に変更**（2026年6月22日時点）。現状の保存先は`config/filesystems.php`の`local`ディスク＝`storage_path('app')`（プロジェクトフォルダ内）のままで問題ないか、コードのデプロイ・フォルダ移動からデータを分離するためプロジェクトフォルダ外の別パスに切り出すべきか**未決定（予定）**。`Storage::`ファサード経由の実装のため、変更時もコントローラー側の修正は不要（`root`設定のみ変更で対応可能）
- 双方向ファイル授受は対応しない（セキュリティ上の判断で確定）
- 自動テスト0件
- 空ファイルアップロードの明示的拒否なし
- 一度もURLに紐づいていないファイルは自動削除対象外
- CSP未導入
- **ユーザー削除時の連鎖削除リスク（要再検討）：** `admin.users.destroy`はユーザーを物理削除し、外部キーの`onDelete('cascade')`設定により、そのユーザーの`shared_files`・`download_urls`（有効期限内のものを含む）・`deleted_files_log`まで連鎖削除される。確認ダイアログ等の警告は無い。退職者整理等での誤操作で取引先・応募者向けの有効なURLが無警告で消える可能性があるため、本番運用前に「ソフトデリート化」「削除前の影響確認画面」「ユーザー無効化（`is_active=false`）への一本化で物理削除自体を提供しない」等の対応方針を検討すること（8.7.1参照）
