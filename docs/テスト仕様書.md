# テスト仕様書

## 本書の目的

打合せで指摘された「受け入れ条件・テスト観点が無い」「何をもって完成かの基準が無い」という課題に対応するため、**現在の実装内容**に基づいて受け入れ条件とテスト観点を整理したもの。

- 自動テスト（PHPUnit等）は本書作成時点で0件。本書は人手での確認手順・チェックリストとして運用する
- 要件定義書・設計仕様書が「あるべき仕様」を記すのに対し、本書は「現状の実装が何を保証しているか」を記す。両者が一致しない箇所は「既知の未対応」として明記する
- 実装が変わった場合、本書を追記・更新すること（このファイル自体は読み取り専用ではない）

## 1. 認証・認可（担当者・管理者）

### 受け入れ条件
- メールアドレス＋パスワードでログインできる
- 無効化（`is_active=false`）されたユーザーはログインできない
- 管理者はURL発行・ファイルアップロードができない（担当者専用機能）
- 担当者は自分が発行したURL・自分がアップロードしたファイルのみ操作できる（他人のリソースはID直接指定でも403）
- 管理者は全データにアクセスできる

### テスト観点

| # | 観点 | 期待結果 |
|---|---|---|
| 1-1 | 正しいメール・パスワードでログイン | ダッシュボードへ遷移 |
| 1-2 | 誤ったパスワードでログイン | エラー表示、ログインできない |
| 1-3 | `is_active=false`のユーザーでログイン | ログイン拒否 |
| 1-4 | 管理者で`/urls/create`にアクセス | 403 |
| 1-5 | 管理者で`/files`のアップロードPOSTを直接送信 | 403（画面上もアップロードUI非表示） |
| 1-6 | 担当者Aが担当者Bの`urls/{id}/edit`等にID指定でアクセス | 403（`authorizeOwner`） |
| 1-7 | 担当者Aが担当者Bのファイルを`files/{id}`で削除 | 403 |
| 1-8 | ログイン済みユーザーが`/login`に再アクセス | `/dashboard`へリダイレクト |

## 2. ファイルアップロード（担当者）

### 受け入れ条件
- 許可拡張子（jpg, jpeg, png, gif, pdf, doc, docx, xls, xlsx, ppt, pptx, zip, csv, txt）以外は拒否
- 1ファイル最大100MB、1回のアップロード合計最大300MBを超えると拒否
- ユーザーのストレージ占有率がしきい値を超えた場合、管理者にメール通知（アップロード自体は継続）

### テスト観点

| # | 観点 | 期待結果 |
|---|---|---|
| 2-1 | 許可拡張子内のファイルをアップロード | 成功、一覧に表示 |
| 2-2 | `.exe`等の非許可拡張子をアップロード | 拒否（422、画面上は「アップロードに失敗しました」） |
| 2-3 | 101MB（1ファイル）をアップロード | 拒否 |
| 2-4 | 合計301MB（複数ファイル）を同時アップロード | 拒否（422、「合計サイズが上限を超えています」） |
| 2-5 | 合計300MB以下・1ファイル100MB以下の複数ファイル | 成功 |
| 2-6 | しきい値超過後に追加アップロード | アップロードは成功し、管理者へ警告メール送信 |
| 2-7 | 警告メール送信が失敗するケース（宛先不正等） | アップロード自体は失敗させない（try-catchで保護） |

## 3. URL発行フロー（3ステップ）

### 受け入れ条件
- 属性（取引先／採用／その他）は必須選択
- 採用選択時、企業名・役職部署欄は非表示・送信時は空になる
- ステップ1→2→3（送付先設定→ファイル選択→メール確認）の順で遷移する
- ファイル選択ステップで新規アップロードと既存ファイルからの選択の両方が可能
- 発行完了画面でメール文章が自動生成され、コピーできる

### テスト観点

| # | 観点 | 期待結果 |
|---|---|---|
| 3-1 | 属性未選択で次へ進む | バリデーションエラー |
| 3-2 | 「採用」選択時に企業名・役職部署を入力済みの状態で送信 | 該当欄は送信時に空扱いになる |
| 3-3 | ステップ2で既存ファイルを選択 | そのファイルがURLに紐づく |
| 3-4 | ステップ2で新規ファイルをアップロード | アップロード後そのファイルがURLに紐づく |
| 3-5 | 発行完了画面で「全文をコピー」 | クリップボードにコピーされる（`navigator.clipboard`不可時は`execCommand('copy')`にフォールバック） |
| 3-6 | 発行完了画面から相手先へ直接メール送信 | 担当者のメールアドレスを送信元として送信される |

## 4. 相手先ダウンロードフロー（認証不要・外部公開）

### 受け入れ条件
- パスコード入力ステップは廃止済み（自動でスキップ扱い）
- メールアドレス確認→OTP（6桁・10分有効）→ダウンロードの順
- メールアドレス確認は15分間に5回まで（それ以上は一時ブロック）
- OTPは5回失敗で30分ロック
- OTP認証完了から10分以内のみダウンロード可能
- ダウンロード回数上限（`download_limit`）に達した場合はダウンロード不可
- 並列リクエストでロックアウト回避・回数上限バイパスができない（行ロックで直列化済み）
- 無効化済み・期限切れ・存在しないトークンは同一の汎用エラー表示

### テスト観点

| # | 観点 | 期待結果 |
|---|---|---|
| 4-1 | 正しいメールアドレス→正しいOTPでダウンロード | ダウンロード成功、`download_count`が1増加 |
| 4-2 | 誤ったメールアドレスを5回連続入力 | 6回目以降は「試行回数が多すぎます」表示（15分間） |
| 4-3 | 誤ったOTPを5回連続入力 | 6回目以降は30分ロック表示 |
| 4-4 | OTP認証完了から11分後にダウンロードURLへアクセス | 再認証が必要（パスコード画面へリダイレクト） |
| 4-5 | `download_limit`到達後に再度ダウンロードを試みる | `download.error`（reason: limit）表示 |
| 4-6 | 同一OTPに対し並列リクエストで5回超の試行を送信 | ロック判定が正しく働く（DB行ロックで直列化） |
| 4-7 | 同一URLに並列リクエストでダウンロードを送信し上限を超えようとする | 上限を超えて加算されない |
| 4-8 | 無効化済みURL／存在しないトークンへアクセス | 同一の汎用エラー画面（`reason: invalid`） |
| 4-9 | 期限切れURLへアクセス | `reason: expired`のエラー画面 |
| 4-10 | `notify_on_download=true`のURLで初回・2回目のダウンロード | 初回のみ担当者へ通知メール送信、2回目以降は送信されない |

## 5. URL管理一覧

### 受け入れ条件
- 検索は「相手先（q）」と「担当者名（staff_q、管理者専用）」で分離されている
- 氏名検索は全角/半角スペースの有無に関わらずヒットする
- 状態（未DL/DL済み/期限切れ/無効化済み）・属性（取引先/採用/その他）は複数選択フィルターで絞り込める
- 各列（相手先・属性・作成日・有効期限・DL数）のヘッダークリックでソートできる
- 無効化済みURLも「無効化済み」フィルターで履歴として確認できる

### テスト観点

| # | 観点 | 期待結果 |
|---|---|---|
| 5-1 | 相手先名に半角スペース有り、検索語はスペース無しで検索 | ヒットする |
| 5-2 | 状態フィルターで「DL済み」のみ選択 | 期限切れかつDL済みの行は含まれない（期限切れ優先） |
| 5-3 | 列ヘッダーをクリック | 並び順が切り替わり、↑↓が表示される。再クリックで昇順/降順が反転する |
| 5-4 | URLを無効化（削除）後に一覧を見る | デフォルト表示でも無効化済みとして表示される（404にはならない） |

## 6. ファイル管理

### 受け入れ条件
- 担当者は自分のファイルのみ、管理者は全件閲覧できる
- ファイル名検索、各列（ファイル名・URL数・アップロード者・サイズ・アップロード日時・削除予定日）でソートできる
- 削除は手動削除（即時・猶予期間の概念なし）と自動削除（cleanup、猶予期間あり）の2系統
- 手動削除すると、紐づくURLもソフトデリートされる
- 管理者にはアップロードUIが表示されない

### テスト観点

| # | 観点 | 期待結果 |
|---|---|---|
| 6-1 | 担当者Aがファイル一覧を開く | 自分のファイルのみ表示 |
| 6-2 | 管理者がファイル一覧を開く | 全担当者のファイルが表示され、アップロード者列が出る |
| 6-3 | 削除予定日でソート | 最も早く削除されるURLに紐づくファイルが先頭/末尾に来る |
| 6-4 | ファイルを手動削除 | 即時に物理削除、紐づくURLが無効化済み扱いになる |
| 6-5 | 管理者が`/files`を開く | ドロップエリアが表示されない |

## 7. 管理者機能

### 受け入れ条件
- ユーザー管理：社員コード・会社ID・部署・権限・状態の追加/編集/削除ができる
- アクセスログ：アクション種別での絞り込み・ページングができる
- ストレージ：ユーザー別内訳・サイズの大きいファイル上位20件が見られる
- 設定：パスコード必須化・自動削除猶予日数・削除前通知・ストレージ警告しきい値を変更できる

### テスト観点

| # | 観点 | 期待結果 |
|---|---|---|
| 7-1 | 新規ユーザー作成（社員コード・会社ID・部署を指定） | 一覧に反映される |
| 7-2 | アクセスログを「失敗のみ」で絞り込む | `email_fail`/`otp_fail`のみ表示 |
| 7-3 | システム設定の猶予日数を変更して保存 | `cleanup_grace_days`が更新され、`fileshare:cleanup`の挙動に反映される |
| 7-4 | 既存のsettings.jsonがある状態で設定画面を保存 | 既存の他項目の値が消えない（読み込んでから上書きする） |

## 8. 自動削除（`fileshare:cleanup`）

### 受け入れ条件
- 有効期限切れ後、猶予期間（デフォルト7日）経過したURLを物理削除する
- 同一ファイルが複数URLで共有されている場合、いずれかのURLがまだ猶予期間内であればファイル本体は削除しない
- アップロードされたが一度もURLに紐づいていないファイルは、このコマンドの対象にならない（既知の制限、9章参照）
- 削除前通知（`notify_before_delete`が有効な場合）は削除予定日の7日前に1回送信
- `deleted_files_log`は1年経過後に自動削除

### テスト観点

| # | 観点 | 期待結果 |
|---|---|---|
| 8-1 | 猶予期間を過ぎたURLがある状態で`fileshare:cleanup`を実行 | URLが物理削除（`forceDelete`）される |
| 8-2 | 1ファイルを2つのURLで共有し、片方だけ猶予期間超過 | ファイル本体は削除されない |
| 8-3 | 両方のURLが猶予期間超過 | ファイル本体も削除され、`deleted_files_log`に記録される |
| 8-4 | 削除予定日の7日前で`notify_before_delete=true` | 担当者に通知メールが送信される |
| 8-5 | 1年以上前の`deleted_files_log`がある状態でコマンドを実行 | 該当ログが削除される |

## 9. 既知の未対応・今後の課題（本書作成時点）

| 項目 | 内容 |
|---|---|
| 自動テストが無い | 本書のテスト観点は手動確認用。CI等への組み込みは未着手 |
| 空ファイル（0バイト）の拒否 | バリデーションに`min`指定が無く、明示的に拒否されない（`docs/ファイル制御仕様書.md`参照） |
| 二重拡張子対策の動作保証 | Laravelの`mimes`ルールの実体バイト列判定に依存しており、意図的なテストケースで確認されていない |
| 孤立ファイルの自動整理 | 一度もURLに紐づいていないアップロード済みファイルは`fileshare:cleanup`の対象外で、手動削除以外に消える手段が無い |
| CSP（Content-Security-Policy）未対応 | インラインstyle多用のため見送り（`SecurityHeaders`ミドルウェア導入時の判断） |
| ウイルスチェック | 未実装（要件定義書に本番リリース前必須と明記） |
