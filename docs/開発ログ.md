## 作業メモ

### 環境構築で発生した問題と解決策

- `httpd-ssl.conf` に `127.0.0.1 oneway-fileshare.local` が混入 → 削除で解決
- Laravelインストール時にPHP 8.4を要求される → `composer.json` に `"platform": {"php": "7.4.10"}` を追加し `composer update` で解決
- Apacheリスンポートを80→8080に変更済み

**2026-06-16**

- KagoyaのSMTPサーバーへの認証が `535 5.7.8 authentication failed` で失敗 → `MAIL_USERNAME` をメールアドレス形式（`y.takano@prime-gr.jp`）からKagoyaのアカウント名形式（`primegroup.ytakano`）に変更して解決
- クリップボードへのコピー機能（`navigator.clipboard`）が動作しない → HTTP環境（非secureコンテキスト）では使用できない仕様のため、`document.execCommand('copy')` へのフォールバックを追加して解決
- `.env` ファイルをエディタで編集・保存しても変更が反映されない問題が発生 → 原因不明（読み取り専用属性ではなかった）。最終的にユーザーの指示を受けてClaude Codeが直接編集して対応

**2026-06-18**

- 認証済みユーザーが`/login`に再アクセスすると404になる不具合 → `RedirectIfAuthenticated`ミドルウェアのリダイレクト先`RouteServiceProvider::HOME`が`/home`（存在しないルート）になっていたため。`/dashboard`に修正して解決
- `fileshare:cleanup`（自動削除）が`Kernel.php`にスケジュール定義されているだけで、実際には一度も実行されていないことが判明 → Windows側に`php artisan schedule:run`を呼ぶタスクスケジューラ設定が無かったため。PCを夜間シャットダウンする運用のため、`schedule:run`の毎分チェック方式ではなく`fileshare:cleanup`を直接1日1回（12:00、`-StartWhenAvailable`付き）実行するタスクをWindowsタスクスケジューラに登録して解決

**2026-06-19**

- ストレージ警告の「DL済み」判定が有効期限切れを考慮しておらず、期限切れ後にダウンロードされたURLまで「DL済み」フィルター・カウントに誤ヒットしていた（テーブル上は「期限切れ」バッジが優先表示されるのと矛盾） → `wait`と同様に`expires_at > now()`の条件を追加して解決
- ストレージ警告メールの送信に失敗すると、ファイルアップロード処理自体が失敗してしまう設計だった（管理者アドレスに実在しないドメイン`admin@example.com`が含まれるため、本番相当のテストで発覚） → メール送信部分をtry-catchで保護し、送信失敗時もアップロードは継続するよう修正
- 通知メールの動作確認時、既存の`settings.json`を読み込まずに上書きしてしまい、デフォルト値で復元する形になった → 復元値はAI_CONTEXT.mdの記載と整合していることを確認したが、ユーザーに「システム設定」画面の再確認を依頼。以後は設定ファイルを直接書き換える前に必ず既存内容を読み込んで復元する方針に変更

**2026-06-22**

- プロジェクトフォルダ移動（`C:\xampp\projects\`→`C:\xampp\htdocs\`）の際、git管理外だった`log_start.bat`/`log_end.bat`・補助の一時ファイルが失われた → Windowsのごみ箱から復元できることを確認。ただし調査の結果2026-06-15以降使われていない廃止済み機能と判明したため、最終的には関連記述も含めて削除する方針に変更
- `DownloadUrlController::complete()`のみ`authorizeOwner()`チェックが欠落しており、ログイン済みの別の担当者が他人のURLのID指定で生ダウンロードトークン入りメール文面を閲覧できる状態だった → ゼロ再構築用仕様書作成のための実装照合中に発覚。`complete()`の先頭にチェックを追加して解決
- ユーザー削除（`admin.users.destroy`）が物理削除であり、外部キーの`onDelete('cascade')`設定により紐づくファイル・URL・削除ログまで連鎖削除されることが判明 → 退職者整理等での誤操作リスクとして仕様書に記録。対応方針（ソフトデリート化・削除前の確認画面・無効化への一本化等）は未着手で今後の検討事項
