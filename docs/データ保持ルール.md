# データ保持ルール

oneway-fileshare（AXON）における各テーブルのデータ保持期間・自動削除の仕組みをまとめたもの。

## 自動クリーンアップの仕組み

`fileshare:cleanup` Artisanコマンド（`app/Console/Commands/CleanupExpiredFiles.php`）が一括して担当し、毎日02:00に自動実行される（`app/Console/Kernel.php`のスケジュール設定）。

処理内容：
1. `expires_at`が過去になっている`download_urls`を全件取得（無効化済み・有効問わず`withTrashed()`）
2. 削除予定日 = `expires_at` + 猶予日数（システム設定の`cleanup_grace_days`、デフォルト7日）を計算
3. 猶予期間が過ぎていれば、その`download_urls`を物理削除（`forceDelete()`）
4. 紐づく`shared_files`が**他の有効なURLからも参照されていないことを確認した上で**、ディスク上のファイル本体・`shared_files`行を削除し、`deleted_files_log`に記録
5. `deleted_files_log`のうち1年（365日）より古いレコードを削除

## テーブルごとの保持ルール

### download_urls（URL発行情報）
- 担当者が「無効化」を押すと**ソフトデリート**（`deleted_at`セット）されるが、行自体はすぐには消えない
- 物理削除されるのは、元々設定された`expires_at` + 猶予日数を過ぎた時点（`fileshare:cleanup`実行時）
- **重要**：無効化した日時は削除タイミングに影響しない。早めに無効化しても、削除は元の有効期限基準で行われる

### shared_files（アップロードファイル）
- 1つのファイルに複数のURLが発行されている場合、**すべてのURLが削除対象（期限切れ＋猶予期間超過）になるまで**ファイル本体・行は削除されない
- ファイル管理画面から手動削除（`FileController::destroy`）した場合は即時削除。紐づくURLはソフトデリートされる（猶予期間の概念なし、即時）

### access_logs（アクセスログ）
- `download_urls`に対する外部キー制約（`onDelete('cascade')`）により、親URLが**物理削除された瞬間にDBレベルで自動的に削除**される
- 独立した保持期間はなく、親URLのライフサイクルに完全に従属する

### auth_codes（OTP認証コード）
- access_logsと同様、親`download_urls`の物理削除に連動してカスケード削除される

### deleted_files_log（ファイル削除履歴）
- 「いつ・誰が・どのファイルを削除したか」の証跡（自動クリーンアップ・手動削除の両方で記録）
- **保持期間：1年（365日）**。`fileshare:cleanup`実行時に1年超過分を自動削除
- 元のファイル名（`original_name`）を含むため、採用カテゴリ等で個人情報が記録される可能性がある点に留意

### users / depts（マスタデータ）
- 自動削除の対象外。永続データとして扱い、管理者による手動操作でのみ変更・削除する

## 関連設定（管理者がシステム設定画面で変更可能）

| 設定キー | デフォルト値 | 内容 |
|---|---|---|
| `cleanup_grace_days` | 7日 | `expires_at`からファイル削除までの猶予期間 |
| `notify_before_delete` | 無効 | 削除予定日の7日前に担当者へ通知メールを送るか |

設定は`storage/app/settings.json`に保存される（DBテーブルではない）。

## 補足

- `deleted_files_log`の1年保持以外のテーブルには、明示的な「経過日数による削除」はない。`download_urls`/`shared_files`の削除はあくまで「発行時に設定した有効期限＋猶予期間」が基準であり、データが古いかどうかでは判断しない
- `access_logs`・`auth_codes`は親テーブルに完全従属するため、このドキュメントでは独立した保持期間を設けていない
